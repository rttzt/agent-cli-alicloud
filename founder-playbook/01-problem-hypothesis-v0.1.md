# Problem Hypothesis v0.1 — Alicloud Agent CLI

> **状态**：Idea Stage / 待用户访谈验证
> **日期**：2026-05-21
> **作者**：Wei Chang（创始人）+ QoderWork（研究伙伴）
> **方法论**：Anthropic Founder's Playbook — Idea Stage

---

## 1. 一句话假设（Falsifiable Problem Hypothesis）

> **国内科技公司（互联网中大厂、SaaS、AI 创业团队）里负责"在阿里云上把 AI Agent 做到生产可用"的研发工程师，每启动或迭代一个 Agent 项目时，平均要跨 ≥4 个割裂的工具与平台（AI 脑暴/共创 → Qoder/Cursor 写代码 → 阿里云百炼控制台/SDK → 函数计算/SAE/PAI 部署 → 自研日志-观测胶水），单次端到端落地通常耗费 5–15 人/天；每次升级模型、换 Agent 框架、改部署形态都要把胶水重新拼一遍。结构性根因是：阿里云生态里目前没有一个面向 Agent 全生命周期（Idea → Design → Code → Eval → Deploy → Observe）的统一 CLI/Skill 层，让 Coding Agent 一次性接管跨产品操作。Google 已用 `agents-cli` 在海外占住了同类心智，国内空白。**

### 1.1 假设的可拆维度

| 维度 | 当前主张 | 需要被验证的数字/事实 |
|---|---|---|
| **WHO（角色）** | 国内科技公司的 Agent 研发工程师（不是普通后端、也不是数据科学家） | 这个角色在受访公司里是否真实存在、是否单独成岗 |
| **WHERE（场景）** | 在阿里云上从 0 → 1 落地生产级 Agent 项目 | 是否真的"必须用阿里云"，还是经常出现混合云/迁移 |
| **HOW OFTEN（频次）** | 每次新项目 + 每次模型/框架升级都会重复 | 平均一年要重拼几次胶水 |
| **HOW SEVERELY（严重程度）** | 单次端到端 5–15 人/天 | 真实数字是多少；是否有人量化过 |
| **TODAY'S WORKAROUND（现状）** | 多工具拼接：脑暴（Claude/QoderWork）+ 编码（Qoder/Cursor/通义灵码）+ 百炼控制台 + FC/SAE 部署 + 自写观测脚本 | 他们是否已经积累了"个人最佳实践"反而不痛了 |
| **WHY NOW（时机）** | (1) Google `agents-cli` 占住了海外样板；(2) 国内 Agent 进入生产爆发期；(3) 通义/百炼 SDK 短期内仍在分裂演进 | 阿里云官方是否已在做同类东西、节奏多快 |
| **STRUCTURAL REASON（结构性根因）** | 阿里云缺一层"面向 Agent 全生命周期 + 面向 Coding Agent 调用 的 CLI/Skill" | 用户是否同意"工具链层"是问题的根，而不是"百炼某个功能"是问题的根 |

### 1.2 用户原话（创始人口述，未润色）

> "用户是科技公司的 Agent 研发工程师……环节割裂，从 idea 到设计、coding、部署、观测涉及非常多不同工具，环节之间断裂、问题不能连贯解决。"
> "他们现在是用不同工具来开发每一个环节——脑暴用 AI 共创、开发用 Qoder、部署用百炼 + 开源框架。"
> "Google 已经推了 `agents-cli`，国内空白，且 Agent 处于爆发期，未来会有越来越多云上 Agent 开发需求。"

## 2. 证伪条件（什么样的访谈结果会让我们必须改方向）

我们承诺：在做完 ≥8 场访谈、其中 ≥5 场来自符合"目标用户画像"的真人后，如果出现下列任意 **2 项以上**，则**立即进入 Pivot 评估**：

- [ ] 多数受访工程师承认割裂，但**已经有自己的 Bash/Makefile/CI 模板**把流程粘合住，不再痛。
- [ ] 多数受访工程师**不愿把"控制权"交给一个 CLI/Coding Agent**——他们更想要文档/样板，而不是更高的抽象。
- [ ] 阿里云官方（百炼 / 通义灵码 / PAI / ModelStudio）在受访期间发布或预告了**同向产品**，且发布节奏 < 6 个月。
- [ ] 用户**承认痛但拒绝付费**（个人工具、公司不立预算、不愿换默认工具链）。
- [ ] 真正最痛的环节其实不是"工具链拼接"，而是**模型选型 / 评测 / 私有数据接入 / 合规** 中的某一项。
- [ ] 受访者反映"我们公司根本不用阿里云做 Agent，我们用 [字节火山 / 腾讯云 / 自建 / 海外]"——目标平台错了。

## 3. Devil's Advocate 报告（8 条最强反驳）

> 这是 Playbook 第 10 页强调的反向用法：用同样的 AI，让它**为我的失败找证据**。
> 任何一条若被访谈证实，都需要在 Idea Stage 内决策。

### R1. 阿里云官方很可能在做（最致命）
百炼已在向"Agent Studio"演进，通义灵码 + ModelStudio + PAI-Agent 都在叠 Agent 能力。一旦官方推出"百炼 CLI / 灵码 Agent Skill"，第三方 CLI 立即失去分发与心智。
**反向假设**：你做的不是"国内空白"，而是"还没轮到的内部路线图"，6–12 个月内会被官方动作整面碾压。
**访谈题**：在你们公司，如果阿里云官方明天发布 `bailian-cli`，你会优先用官方还是第三方？为什么？

### R2. "工具链割裂"可能是**一次性痛**而非**持续痛**
工程师跑通第一个 Agent 后，往往会把脚本和模板沉淀进自己的 repo / 公司内部脚手架，第 2 个项目复用率极高，不再需要外部 CLI。
**反向假设**：你瞄准的是"第一次"的痛，不是高频痛——天花板很低。
**访谈题**：你们组上一次新建 Agent 项目是什么时候？这次和上次相比，重复劳动的比例有多高？

### R3. ICP 太窄 + 不付费
"国内 Agent 研发工程师"全国可能仅 1–3 万人；他们个人不掏钱买开发工具，公司预算又走平台部统采。开源项目大概率被 fork / 白嫖。
**反向假设**：这是个"GitHub Star 项目"，不是"商业级产品"。
**访谈题**：在你们公司，开发工具的预算从哪一行预算里出？你个人最近一次为开发工具付费是什么？

### R4. Coding Agent 生态分裂，且阿里云用户主要用通义灵码
国内阿里云大客户更可能默认装通义灵码，灵码自己一定会做 Agent 能力。Qoder/Cursor/Claude Code 在国内大公司渗透率远不如灵码。
**反向假设**：分发渠道堵死——即使你产品做得好，用户也用不上。
**访谈题**：你们组主力 Coding Agent 是什么？换其他工具需要走什么流程？

### R5. "Google agents-cli 国内空白" 是 cargo cult
Google `agents-cli` 在国内开发者渗透率本身近 0，"空白"反映的是 GCP 在中国份额低，而不是"国内开发者需要这种工具"。把海外结论直接平移到阿里云，是论证错位。
**反向假设**：参照对象错了——你应该参照的不是 Google `agents-cli`，而是 Dify / FastGPT / Coze 等国内 Agent 平台的接入路径。
**访谈题**：你听说过 `google/agents-cli` 吗？如果有阿里云版你会装吗？为什么？

### R6. "Agent 爆发期" 是宏观叙事，不是具体痛点
爆发期 = 大量短命实验项目；工程师对"实验最快"的兴趣，远高于对"标准化全流程"的兴趣。CLI 这种"建模流程"的产品，在爆发期反而是逆风。
**反向假设**：现在该做的是"实验加速器"，不是"全生命周期标准化器"——产品形态错了。
**访谈题**：过去 3 个月你立过的 Agent 项目里，有几个上了生产、几个废弃？废弃的为什么废弃？

### R7. 维护成本会吃掉一切
阿里云百炼 / 通义 / FC / SAE 每季度都会出 break change。第三方 CLI 每次都要追适配，少数人维护根本不动；用户跑一次发现版本不对，就再也不回来。
**反向假设**：这不是产品问题，而是**运维债务问题**——你需要的不只是建个 CLI，而是一支能跟住阿里云迭代节奏的工程团队。
**访谈题**：你上次因为阿里云 SDK 升级踩过什么坑？你愿不愿意依赖一个非官方 CLI 来抹平这种坑？

### R8. 你是 ICP 本身——这是双刃剑
你自己是这个角色，所以痛感真实；但你也最容易**用自己的痛代替市场的痛**，并且从身边几位同事身上"看到证据"。
**反向假设**：你描绘的可能不是市场，而是 5–10 个跟你工作模式相似的人。
**访谈题**（在内部要逼自己回答）：除你之外，能立刻报出名字的 8 位"我确定他和我一样痛"的人是谁？他们公司彼此重合度多高？

## 4. 第一版"目标用户精确画像"

| 字段 | 候选值（待访谈收敛） |
|---|---|
| 公司类型 | 互联网中大厂 BU / AI 应用创业公司（A 轮前后）/ 阿里云生态 ISV |
| 团队规模 | 5–30 人技术团队，其中 1–3 人专做 Agent |
| 职位 | Agent 研发工程师 / AI 应用工程师 / 大模型应用算法工程师 |
| 司龄 | 1–5 年（既懂工程又懂 AI 落地，而非纯算法或纯后端） |
| 必备特征 | 过去 6 个月在阿里云（百炼 / FC / SAE / DashScope）上至少完整跑通过 1 个 Agent 项目 |
| 排除项 | 纯学术 / 纯调用第三方 SaaS（Dify、Coze）从不写代码 / 完全用海外云的团队 |
